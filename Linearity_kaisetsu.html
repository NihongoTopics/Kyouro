Linearity of Meaning の内容と意義

Claude により生成。／ Semantics of Tokyo Japanese の続き。教育ローマ字を少し知っている人向けのまとめ。

この文書は2つの部分でできている。第1部 Types は用語の改訂で、形式体系そのものは変えない。第2部 Linearity は、Semantics に出てきた意味の基底構造を1つの線形パターンに一般化する提案。以下、順番に見る。

第1部　Types（用語の改訂）

出発点は Semantics の Convention 4.1。中身はこう書かれている。

Convention 4.1
role は直前の serial を取って、role つきの serial を作る。role のある serial とない serial は別のタイプとして扱う。parallelizer が取れるのは role のある方だけ。

angle bracket ⟨ ⟩ は「中身のタイプを返す関数」。Convention 4.1 の要点は3つ。

  • ⟨X⟩ と ⟨R⟩ がこの順で隣り合うと merge できる。
  • merge すると ⟨RX⟩ になる。
  • parallelizer が左向きに取れるのは ⟨RX⟩ の方で、⟨X⟩ は取れない。
\[ \langle X\rangle \quad \langle {}_R\rangle \qquad \langle {}_R X\rangle \;=\; \left\langle \frac{{}_R A \cdot_R X}{\,{}_R B \cdot_R Y\,} \right\rangle \;\neq\; \langle X\rangle \]

旧用語では、⟨X⟩、⟨RX⟩、そして上の分数形（vinculum を持つ形）を、3つとも区別せず serial と呼んでいた。でも Convention 4.1 が言うとおり、⟨X⟩ だけは扱いが違う（parallelizer に取ってもらえない）。同じ名前だと、この違いが名前から見えない。そこで名前を分ける。

  • ⟨X⟩ を bud と呼ぶ。
  • ⟨RX⟩ を flower と呼ぶ。
  • 分数形 \({}_R A\cdot_R X \,/\, {}_R B\cdot_R Y\) を cyme と呼ぶ。
  • serial は flower と cyme をまとめて指す言葉。bud は含まない。

parallelizer と parallel の定義は Semantics から引き継ぎ。parallelizer は「左に記号＋コロン＋両側に記号を持つ vinculum」の形。例は \(\langle X : \frac{\text{iscute}}{\lnot\text{iscute}}\rangle\) で、タイプは [S : P]。parallel は「両側に記号を持つ vinculum」で、例は \(\langle \frac{\text{iscute}}{\lnot\text{iscute}}\rangle\)、タイプは P。parallelizer は serial を取って parallel を作る。serial は mood M と merge でき、clause の末尾で root node ROOT ができる。

Rule 1.1（clause 内の merge 制約）

記号だけで書くと次の4本。\(X, Y : Z\) は「X, Y がこの順で merge して Z になる」の意味。

\[ B, {}_R : S \qquad S,\, [S:P] : P \qquad P, P : S \qquad S, M : \text{ROOT} \]

merge は入力ノードを結果ノードでその場（in situ）に置き換える。これらのルール自体は新しくない。Semantics に既に出ている（Convention 4.1 を採用する前の形で書かれていたもの）。

第1部で実際に変わったのは名前だけ。形式体系は変わらない。ポイントは、merge 規則に現れる区別（bud か serial か）と、現れない区別（flower か cyme か。この2つは同じタイプ S）を、名前のレベルで整理したこと。flower と cyme が同じ S なので、その違いは merge 規則には出てこない。

第2部　Linearity

ここが新しい提案。東京日本語で観察された意味基底構造を、形式的に一般化する。著者ははっきり限定をつけている。「この一般化は言語の性質の一部を反映していると期待するが、どの範囲まで成り立つかの証拠は今の時点では出さない」。

Definition 2.1　Rhipidial clause

Semantics に出てくる clause は全部、次の線形パターンになっている。上付きの変数は繰り返し回数。

\[ \big[\, [B, {}_R, [S:P]]^{\,i},\; B, {}_R, [S:P]^{\,i},\; M \,\big] \]

つまり、[B, R, [S:P]] が i 回開いて、真ん中に B, R があり、右側に [S:P] が i 回閉じて、最後に mood M が来る。この形の clause を rhipidial clause と呼ぶ。

rhipidium は植物学の用語で、枝が同じ平面に並ぶ扇形の cyme（集散花序）のこと。bud / flower / cyme という語もこの体系がすでに使っている植物系の語で、rhipidial はその線上にある命名。

すべての rhipidial clause は Rule 1.1 を満たす。ただし逆は成り立たない。そして、東京日本語の文に non-rhipidial な clause が出てくるかどうかは、まだ分かっていない（未解決の問題として置かれている）。

関係節と index mood

1つの文が複数の clause からできることもある。関係節（relative clause）の root node は、index mood という特別な mood が serial と merge して得られる。それがさらに bud と merge して bud を作り、その bud が外側の clause の一部になる。関係節まわりの規則は Semantics の Convention 4.2（index role の規則）として与えられていて、index の番号が一致するときだけ bud ができ、番号が合わないときや index でない role のときは merge が失敗する、という形になっている。

Rule 2.1（sentence 内の merge 制約）

Rule 1.1 と Convention 4.2 をまとめると、文レベルの merge 制約が得られる。Rule 1.1 の4本に、関係節を外側に取り込む1本が加わる。

\[ B, {}_R : S \qquad S,\, [S:P] : P \qquad P, P : S \] \[ S, M : \text{ROOT} \qquad \text{ROOT},\, B : B \]

最後の ROOT, B : B が、関係節の root と後続の bud が merge して bud になる規則。

Definition 2.2　Rhipidial sentence

Rule 2.1 を使うと、全部の clause が rhipidial であるような文を定義できる。それが rhipidial sentence。文の意味基底は次の線形構造。上付きの変数は繰り返し回数。

\[ \big[\, [B, {}_R, [S:P]]^{\,i},\; B, {}_R, [S:P]^{\,i},\; M \,\big] \]

この中に出てくる bud B は、それ自体を次の線形構造で置き換えてもよい。置き換える文字列のほうは、末尾に B が付く。

\[ \big[\, [B, {}_R, [S:P]]^{\,j},\; B, {}_R, [S:P]^{\,j},\; M,\; B \,\big] \]

変数 i, j と B について、同じ規則が再帰的に当てはまる。つまり関係節を通して、rhipidial な構造が bud として文の中に入れ子で埋め込まれ、それが再帰する。

この文書の意義

文書から実際に言えることを整理すると、こうなる。

  • 第1部：名前の改訂によって、merge 規則で別扱いされる区別（bud と serial）が名前から見えるようになった。旧用語は3つ全部 serial で、この区別が名前に出ていなかった。flower と cyme は同じタイプ S なので、その違いは merge 規則には現れない。
  • 第2部：Semantics で記述された clause が、例外なく1つの線形パターン（rhipidial）に収まる、という一般化。clause も sentence も同じ扇形の型で書け、関係節の再帰で入れ子になる。merge 規則も clause 用の Rule 1.1 と sentence 用の Rule 2.1 に統一される。
  • 留保：著者自身が「non-rhipidial な clause があるかは未知」と書いている。だから主張は「Semantics で記述した範囲は全部この形」であって、「東京日本語の全部の文がこの形」ではない。
